## 1. What the Confluence Network Is

The Confluence Network is a fractal reasoning architecture in which queries are routed through a recursive hierarchy of specialised pods, processed at the depth appropriate to their complexity, and returned with progressively integrated results.

It has three layers of description, each necessary, none alone sufficient:

| Layer | Description | Best representation |
|-------|-------------|-------------------|
| **Routing logic** | How queries travel, branch, and return | Routing diagram (IMG_7632) |
| **Topology** | How nodes connect across the network | Square grid / torus |
| **Geometry** | The underlying fractal shape | Sierpinski tetrix |

These are not three different systems. They are three views of one system.

## 2. The Three-Layer Architecture

The Confluence Network has a clean separation of function across three layers:

| Layer | Name | Function |
|-------|------|----------|
| **Confluence Network** | Routing architecture | Where reasoning occurs — routes queries to appropriate pods at appropriate depth |
| **Pods** | Reasoning units | Who does the reasoning — four complementary roles, fully connected |
| **GPSL** | Reasoning grammar | How reasoning proceeds — symbolic operator language running inside each pod |

GPSL and the Confluence Network are complementary but independent. GPSL can be used without the network. The network can route any reasoning process. Together they form a complete architecture for distributed symbolic reasoning.

## 4. The Three Knowledge Pillars

The first level of the network divides all incoming queries into three primary domains — a triadic structure rooted in one of the most stable epistemological frameworks in human thought.

Francis Bacon formalised this division in 1605 in *Novum Organum*:

| Faculty | Knowledge domain |
|---------|-----------------|
| Memory | History |
| Reason | Philosophy |
| Imagination | Poetry / Creative expression |

This framework has survived across cultures and centuries because it maps something real about how minds process information. It is not arbitrary. The Confluence Network adopts it not as homage but because the geometry demands exactly three primary subdivisions, and this is the most robust three-part division of all human knowledge available.

**The prime node** (O) sits above the three pillars. It carries no domain assumption — pure triage. Every query enters here. Simple queries resolve here. Complex queries are routed to one or more pillars.

## 5. Node Types and Their Roles

The tetrahedron renders as four distinct node types in 2D:

**Prime Node (O / Θ)**
- Single entry point for all queries
- No domain assumption — pure triage function
- Returns final integrated answer on the way back up
- Structurally distinct: open circle, belongs to no domain

**Pillar Nodes (solid squares)**
- Memory / History
- Reason / Philosophy
- Imagination / Poetry
- Each is the root of its own sub-network — a complete Confluence Network in miniature

**Shared Connection Nodes (◑)**
- Dual membership — belong simultaneously to both adjacent pods
- Not connectors *between* pods; nodes that *are* both pods at that point
- Primary cross-check points on descent and return
- Agreement = confidence signal; disagreement = flag for deeper processing

**Θ Integrator (central node)**
- Receives signals from all three pillars
- Holds the full picture before the answer returns to the prime node
- Sees everything, belongs exclusively to nothing

## 6. Routing Logic

### Descent

A query enters at the Prime Node. Triage assesses complexity and domain:

- **Simple / general** → resolved at Prime Node level, no routing
- **Domain-specific** → routed to appropriate Pillar Node via shared connection node
- **Cross-domain** → routed to two or more Pillars simultaneously, parallel paths

At each Pillar Node, the same triage logic applies recursively. The query descends through layers of increasing specialisation until consensus can form. Depth varies by problem — a surface question resolves at L1, a deep philosophical problem may descend to L4 or L5.

### The Zoom Operation

The central routing mechanism is the **zoom operation**. Any pod node can be expanded — it becomes the Prime Node (O) for the cluster one level below.

> The node does not change. It does not split or replicate. It is simultaneously a domain node in the cluster above and the prime node of the cluster below.

This is *recursive without duplication*. Most hierarchical systems implement depth by copying nodes into subtrees. The Confluence Network preserves node identity across scales. One node, two roles, two levels. The dashed border in the routing diagram is the visual marker of this dual identity.

### Cross-Checking on Descent

As queries branch through sub-domains, paths converge at shared connection nodes. At each convergence point, partial results from different branches are compared. This is not a designed quality-control step — it is a geometric inevitability. The architecture enforces cross-checking without it needing to be specified.

### Return Journey

Consensus forms at depth and propagates back up. The return journey mirrors the descent — same nodes, same shared connection points, but carrying confirmed results rather than open queries.

At each shared node on the way back up, returning signals from parallel branches meet and combine. Progressive integration — not a single synthesis at the top, but continuous convergence as the answer assembles itself on the way home.

Final integration happens at Θ before the complete answer returns to the Prime Node.

## 7. The Process Fractal

The Confluence Network is not a static shape. It is a process that happens to have the shape of a Sierpinski tetrix.

The fundamental process is:

> Input received → roles engage → Θ integrates → output emitted → feedback propagates

This process runs at every layer simultaneously. At the node level, individual roles process a cipher element. At the pod level, four nodes process a full cipher. At the cluster level, four pods process an integrated signal.

The grammar does not change between levels. The operators do not change. What changes is the scale of the unit being processed and the richness of accumulated context.

*As above, so below — not as metaphor, but as architectural specification.*

### Continuous Operation

The network does not pause between user interactions. The interior process is continuous. What a user interaction does is not start the process — it surfaces a particular moment of the ongoing process into human-legible form.

## 8. The Rotating Roster

Each interaction is handled by a designated Input Pod selected from the available pool. When a new Input Pod is selected, the entire hierarchical structure beneath it reorganises accordingly.

This is geometrically coherent because the Sierpinski tetrix is self-similar from any node. Any pod can be root without changing the structure. The rotation exploits a property the geometry already has.

What the rotation achieves:
- No single pod accumulates disproportionate context
- Load distributes without centralised routing logic
- Each user surfaces the process at a different point, producing genuinely varied outputs from the same network state

## 9. The Two Physical Representations

### Sierpinski Tetrix
Best for understanding: depth, specialisation, fractal self-similarity, the breathing dynamic.

### Square Grid (rotated 45°)
When the tetrix is unfolded into its connection graph, the result is a square grid. Each diamond is a pod. Adjacent diamonds share corner points exactly.

Best for understanding: routing, topology, uniformity, scalability.

The square grid is mathematically equivalent to a **2D torus** — each node connected to exactly four neighbours, vertex-transitive, rotation-invariant. The topology problem (what is the optimal inter-pod connection structure for the rotating roster?) is answered by the geometry: the tetrix already has it.

## 10. Relationship to GPSL

GPSL runs *inside* each pod. The Confluence Network routes queries *to* pods. They are independent and complementary:

```
Query enters Prime Node
        ↓
Confluence Network routes to appropriate pod at appropriate depth
        ↓
Pod receives query as GPSL cipher (or converts to cipher)
        ↓
GPSL grammar describes reasoning inside the pod
        ↓
Pod emits result
        ↓
Confluence Network routes result back up through convergence nodes
        ↓
Final answer returns to Prime Node
```

The header in GPSL corresponds to the routing level in the Confluence Network. A query routed to `Reason / Philosophy / Ethics` activates `HEADER: Ethics` inside the pod that receives it. The network and the grammar use the same semantic structure — domain hierarchy — from two different directions.
