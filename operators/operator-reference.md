# Operator Reference

## Core Operators

| Operator | Name | Function | Notes |
|---|---|---|---|
| `→` | Transition | Process-to-process transformation | Core directional operator |
| `=` | Assignment / resolution | Resolves an expression into an output | Accepts valid compound expressions on the left |
| `⊗` | Entanglement / composition | Couples symbolic elements | Often increases relational complexity |
| `::` | Bridge | Structural linkage or phase boundary | Used for connection without direct transformation |
| `↺` | Loop / feedback | Recursive return or self-reference | Key for feedback structures |
| `;` | Parallel | Co-present expressions | Separates simultaneous or independent paths |

## Modifiers

| Operator | Name | Function |
|---|---|---|
| `↑` | Amplification | Indicates strengthening, escalation, or intensification |
| `↓` | Dampening | Indicates attenuation, inhibition, or reduction |

## Node Markers

| Form | Type | Function |
|---|---|---|
| `[A]` | Process node | Action, transformation, operation |
| `{A}` | State node | Condition, field, stabilized state |

## Emergent Operators

### `⟶` — Meta-transition

Observed function:
- transition between cipher stages
- transition between representational levels
- meta-level progression

Status:
- empirically recurrent
- not yet treated as fully canonical core syntax

## Operator Admission Principle

An operator should not be admitted because it is familiar. It should be admitted only if it:

1. survives repeated use without destabilizing the system
2. preserves recursive closure
3. avoids introducing ambiguity or drift
4. emerges independently across runs or models

This principle follows from experimental results rather than design preference.
